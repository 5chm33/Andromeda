# Andromeda v2: SOTA Enhancement Roadmap
**Targeting >85% RSI Commit Success Rate**

This roadmap outlines the next tier of State-of-the-Art (SOTA) enhancements designed to push the Andromeda Recursive Self-Improvement (RSI) engine from its current ~65-75% success rate to >85%. 

With the recent implementation of the 3-strategy TypeScript Heal Engine, scope-limited `tsc`, brace-balancing guards, and the multi-model consensus abstention mechanism, the foundation is highly stable. The next phase focuses on proactive context gathering, advanced rollback mechanisms, and dynamic validation.

## 1. Multi-Agent Proposal Generation (Debate & Refine)
Currently, proposals are generated by a single LLM pass and then voted on by the consensus engine. To increase the quality of the initial proposal:
*   **Implement an Actor-Critic architecture:** Before a proposal is submitted to the quick validator, an internal "Critic" agent reviews the code for logic flaws, security vulnerabilities, and architectural consistency.
*   **Iterative Refinement:** If the Critic flags issues, the Actor refines the proposal. Only proposals that pass the Critic's review proceed to the consensus vote.

## 2. Dynamic AST-Based Context Injection
The current TS Heal Engine injects a fixed 40-line window around errors. This can be improved by:
*   **AST Parsing (Abstract Syntax Tree):** Use the TypeScript compiler API to parse the AST and extract exactly the relevant interfaces, types, and function signatures associated with the error, regardless of where they are defined in the file.
*   **Cross-File Context:** If an error involves an imported type, automatically fetch and inject the definition of that type from the external file.

## 3. Sandboxed Execution Verification (Dry-Run Mode)
Before applying changes to the live filesystem and running the full test suite:
*   **Ephemeral Sandboxes:** Spin up lightweight, isolated environments (e.g., using `vm2` or isolated worker threads) to execute the modified code in a dry-run mode.
*   **Behavioral Assertions:** Run targeted micro-tests against the modified functions to ensure they meet expected behavioral contracts before committing to the full CI run.

## 4. Semantic Rollback & Dependency Graphing
Currently, rollbacks revert files to their previous state. This can cause cascading failures if multiple proposals touch interconnected files.
*   **Dependency-Aware Rollbacks:** Maintain a real-time dependency graph. If a rollback is triggered, automatically identify and revert any subsequent proposals that depended on the reverted changes.
*   **Semantic Merging:** Instead of pure file replacement, use semantic merging to revert only the specific AST nodes introduced by the failing proposal, preserving unrelated concurrent changes.

## 5. Automated E2E Visual Regression Testing
For UI changes, `tsc` and unit tests are insufficient.
*   **Playwright/Cypress Integration:** Integrate automated headless browser testing to capture screenshots of modified UI components.
*   **VLM (Vision-Language Model) Judging:** Pass the "before" and "after" screenshots to a VLM (like GPT-4o) to verify that the UI changes match the intent of the proposal and haven't introduced visual regressions (e.g., overlapping elements, broken layouts).

## 6. Continuous Reinforcement Learning from AI Feedback (RLAIF)
*   **Dynamic Weighting:** The consensus engine currently uses static weights for different models. Implement a system where models are dynamically re-weighted based on their historical accuracy in approving successful proposals vs. rejecting failed ones.
*   **Automated DPO Dataset Generation:** Automatically curate a dataset of "successful proposals" vs "failed proposals" and use it to periodically fine-tune a local LoRA model dedicated specifically to the Quick Validator role.

## Conclusion
Implementing these SOTA features will transform the RSI engine from a reactive system (fixing errors after they occur) to a proactive, highly resilient intelligence capable of sustained, complex architectural evolution.
