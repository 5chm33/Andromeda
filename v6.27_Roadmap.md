## Track A: Fix RSI So It Actually Applies Changes

These are the five issues identified from the v6.26 runtime data. They must be fixed before RSI can produce real improvements.

### A1 — Proposal Deduplication `v6.28` ⚡ HIGHEST PRIORITY

**Problem:** 45% of the 282 proposals in the store are duplicates. The same fix is regenerated 5–14 times per cycle because the generator has no memory of what it already proposed.

**Fix:** In `selfImprove.ts`, before inserting a new proposal, compute a hash of `targetFile + title` (or `targetFile + diff[:200]`) and skip if it already exists in the store. Add a `seenHashes: Set<string>` to the in-memory proposal store.

**Files:** `server/selfImprove.ts`  
**Effort:** ~30 lines

---

### A2 — Confidence Score Population `v6.28` ⚡ HIGH PRIORITY

**Problem:** `proposal.confidence` is always `null`. The `minConfidenceThreshold: 0.8` filter in `rsiEngine.ts` is a no-op — it passes everything through.

**Fix:** In the proposal generation LLM prompt, add: *"After the diff, output a JSON object: `{"confidence": 0.0–1.0, "reasoning": "one sentence"}`."* Parse this and store it in `proposal.confidence`. The RSI engine's existing threshold filter will then actually work.

**Files:** `server/selfImprove.ts`, `server/rsiEngine.ts`  
**Effort:** ~50 lines

---

### A3 — Constitution-Aware Proposal Generation `v6.28`

**Problem:** The proposal generator writes diffs that reference `process.env.DEEPSEEK_API_KEY` directly, which the Constitution immediately blocks. The generator doesn't know the rules.

**Fix:** Inject the Constitution's forbidden patterns into the LLM system prompt used for proposal generation. Pull the forbidden list from `safetySupervisor.ts` at runtime so it stays in sync automatically.

**Files:** `server/selfImprove.ts`, `server/safetySupervisor.ts`  
**Effort:** ~20 lines

---

### A4 — File-Aware Proposal Generation `v6.28`

**Problem:** Proposals hallucinate import paths (e.g., `../drizzle` in `biasDetector.ts`) because the generator doesn't read the actual current file content before writing diffs.

**Fix:** The RSI proposal generation step must call `readFileSync(targetFile)` and include the actual current file content in the LLM context window before asking for a diff. This is already the documented workflow in `ANDROMEDA.md` — it just needs to be wired into the automated cycle.

**Files:** `server/selfImprove.ts`, `server/rsiEngine.ts`  
**Effort:** ~40 lines

---

### A5 — Re-capture Eval Baseline `v6.28` (config, not code)

**Problem:** The stored baseline of 2% is invalid — every task returned 401 because the API key was misconfigured at the time. The RSI proof pipeline (`GET /api/rsi/proof`) compares all future scores against this meaningless number.

**Fix:** Set a valid API key in `.env`, then call `POST /api/eval/baseline` once. This will run the 13-task benchmark suite and store the real starting score. No code changes needed.

**Impact:** Unlocks the RSI proof pipeline and gives you a real baseline to beat.

---

### A6 — RSI Cycle Reporting `v6.29`

**Problem:** The RSI history only has one entry, and it contains minimal information about why proposals were rejected (just a count of 129 rejections with no breakdown).

**Fix:** Enhance `rsiEngine.ts` to log per-proposal outcomes to `rsi-history.jsonl`: which proposals were attempted, which passed/failed, what the error was, and what the confidence score was. This makes the RSI loop observable and debuggable.

**Files:** `server/rsiEngine.ts`  
**Effort:** ~60 lines

## Track B: Close the Agent Capability Gap

These are the features where Andromeda currently lags behind Manus and Devin.

### B1 — AST-Based Semantic Chunking `v6.29`

**Problem:** The RSI engine uses line-based file chunking (`fileEngineChunking.ts`). When a refactor requires changing a function signature in File A and all its callers in Files B, C, D, the line-chunker loses the cross-file context and generates incomplete diffs.

**Fix:** Integrate Tree-sitter (already available as an npm package) to parse TypeScript files into an AST. Chunk by function/class boundaries rather than arbitrary line counts. This means a "chunk" is always a complete, semantically meaningful unit.

**Files:** `server/fileEngineChunking.ts`, new `server/astChunker.ts`  
**Effort:** ~200 lines + Tree-sitter integration

---

### B2 — Multi-Agent Parallel Dispatch `v6.29`

**Problem:** `spawn_sub_agent` exists but is never called automatically. The task planner generates sequential plans even when steps are parallelizable (e.g., "Research Topic A" and "Research Topic B" could run simultaneously).

**Fix:** In `taskPlanner.ts`, add a `detectParallelizable(plan: TaskPlan): PlanStep[][]` function that identifies steps with no dependency edges between them. In `reactEngine.ts`, dispatch these groups to sub-agents via `spawnSubAgent` and merge results before the next sequential step.

**Files:** `server/taskPlanner.ts`, `server/reactEngine.ts`, `server/spawnSubAgent.ts`  
**Effort:** ~150 lines

---

### B3 — Vision-First Browser Automation `v6.29`

**Problem:** `browserClick` uses CSS selectors as the primary strategy. Modern React/Next.js apps use dynamic class names (e.g., `css-1x2y3z`) that break constantly. The vision fallback added in v6.23 exists but is secondary.

**Fix:** Flip the default: take a screenshot first, send it to the vision model with "identify the element to click", get back coordinates, click by coordinate. Fall back to CSS selectors only if vision fails. This is how Manus operates.

**Files:** `server/tools/browserTools.ts`, `server/browser.ts`  
**Effort:** ~80 lines

---

### B4 — Cross-File Refactoring Awareness `v6.30`

**Problem:** When the RSI engine modifies a function signature in one file, it does not automatically update callers in other files. This causes TypeScript errors that the type-check step catches and rolls back, making the change impossible to apply.

**Fix:** Before applying a proposal that changes a function signature, use `dependencyGraph.ts` (already built) to find all callers. Generate a multi-file proposal that updates the signature and all call sites atomically. Use `selfModifyBatch` (already built) to apply them as a single transaction.

**Files:** `server/rsiEngine.ts`, `server/dependencyGraph.ts`, `server/selfModify.ts`  
**Effort:** ~100 lines

---

### B5 — Streaming Token Budget Per Session `v6.30`

**Problem:** All agent sessions share a global token budget. A long-running session can starve other sessions.

**Fix:** Implement per-session token budgets in `reactEngine.ts`. Each session gets an isolated counter. When a session hits its budget, it gracefully summarizes and continues rather than hard-stopping.

**Files:** `server/reactEngine.ts`, `server/agentStateMachine.ts`  
**Effort:** ~60 lines

## Track C: Infrastructure

### C1 — Postgres Migration for RSI State `v6.30`

**Problem:** RSI history, proposals, episodic memory, and eval results are stored as flat JSON files. This works for a single instance but breaks under concurrent access and makes horizontal scaling impossible.

**Fix:** Move the following to Postgres via Drizzle (schema already exists):
- `workspace/.andromeda_proposals.json` → `proposals` table
- `workspace/rsi-history.jsonl` → `rsi_cycles` table  
- `workspace/evals/eval-history.jsonl` → `eval_runs` table
- `workspace/memory/episodes.jsonl` → `episodes` table

**Files:** `server/selfImprove.ts`, `server/rsiEngine.ts`, `server/episodicMemory.ts`, `server/evalFramework.ts`  
**Effort:** ~300 lines + Drizzle migrations

---

### C2 — Redis for Rate Limiting and Locks `v6.31`

**Problem:** The per-file mutex in `selfModify.ts` uses in-process `Map<string, Promise>`. This works for a single process but is invisible to other Node.js workers or Docker replicas.

**Fix:** Replace the in-process mutex with a Redis distributed lock (using `ioredis` + `SET NX PX`). Replace the in-memory rate limiters with Redis counters.

**Files:** `server/selfModify.ts`, `server/security.ts`, new `server/redisLock.ts`  
**Effort:** ~100 lines + Redis dependency

---

### C3 — GitHub Actions: Auto-Deploy on RSI Apply `v6.31`

**Problem:** When RSI applies a change, it commits to git but there is no automated deployment step. The running server must be manually restarted.

**Fix:** Extend `.github/workflows/ci.yml` to trigger on pushes to `main`. On a successful CI run, SSH into the deployment server and run `pm2 reload andromeda`. This closes the loop: RSI applies → CI validates → server auto-updates.

**Files:** `.github/workflows/ci.yml`, new `.github/workflows/deploy.yml`  
**Effort:** ~50 lines of YAML
